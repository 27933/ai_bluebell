## 上下文

Bluebell 是一个基于 Go/Gin 的博客系统，包含文章发布、评论、点赞、栏目管理、标签管理等功能。系统运行在 Docker 容器中，使用 MySQL 作为主数据库，Redis 作为缓存。

当前状态：
- 已有 40+ 个 API 接口实现完整
- 仅有少量单元测试（如 jwt_test.go, like_test.go）
- 缺少端到端的 API 集成测试
- 代码变更依赖人工验证

## 目标 / 非目标

**目标：**
- 建立完整的 API 集成测试套件
- 覆盖所有公开接口（无需认证）、认证接口、管理员接口
- 验证正向场景和异常场景
- 支持在 Docker 容器中执行测试
- 提供测试执行脚本和文档

**非目标：**
- 不修改业务逻辑代码（仅添加测试）
- 不引入新的测试框架（使用 Go 标准 testing 包 + testify）
- 不提供性能测试（仅功能测试）
- 不覆盖前端代码

## 决策

### 1. 测试目录结构
**决策**: 使用 `tests/api/` 目录存放所有 API 集成测试
**理由**:
- 与单元测试（`*_test.go` 同级目录）物理分离
- 保持项目结构清晰
- 便于独立运行集成测试
- 测试代码量较大，单独目录更合理

**替代方案**: 放在 `test/` 目录（太通用）或与源码同级（`controller/api_auth_test.go`）

### 2. 测试框架选择
**决策**: 使用 Go 标准 `testing` 包 + `testify/assert`
**理由**:
- 项目已有 testify 依赖
- 标准库无需额外学习成本
- testify 提供丰富的断言方法

### 3. 测试数据准备
**决策**: 使用 setup/teardown 模式，在测试中创建和清理数据
**理由**:
- 避免依赖外部 fixtures 文件
- 测试之间数据隔离
- 更清晰的数据依赖关系

### 4. HTTP 客户端
**决策**: 使用标准 `net/http` 包，封装 HTTP 请求工具函数
**理由**:
- 无需引入 gin 的测试包
- 更接近实际 HTTP 调用
- 灵活性更高

### 5. 环境配置
**决策**: 从 `conf/dev.yaml` 读取配置，支持环境变量覆盖
**理由**:
- 与生产配置保持一致
- 支持 Docker 环境
- 避免硬编码配置

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| 测试数据污染生产数据库 | 使用独立的测试数据库或在 setup 中清空相关表 |
| 测试执行时间过长 | 支持并行测试执行，合理分组测试用例 |
| JWT Token 过期导致测试失败 | 在测试中使用长效 Token 或每次测试前刷新 Token |
| 接口变更需要同步更新测试 | 建立 API 变更时必须更新测试的流程规范 |
| 环境依赖（MySQL/Redis）未就绪 | 在执行测试前添加健康检查 |

## 迁移计划

不需要迁移，这是新增功能。

部署步骤：
1. 创建测试目录结构和文件
2. 编写测试用例
3. 在 Docker 容器中验证测试执行
4. 修复失败的测试用例
5. 更新开发者文档
