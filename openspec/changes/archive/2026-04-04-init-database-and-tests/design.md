## 上下文

Bluebell 是一个博客平台，使用 Go + Gin 框架开发。本地开发环境使用 Docker 运行 MySQL 8.0 和 Redis。服务监听 8084 端口。

当前状态：
- 服务代码已完整，包含完整的 DAO 层实现
- 测试框架已就绪（`tests/api/`）
- 数据库为空，无表结构
- SQL 目录不存在

## 目标 / 非目标

**目标：**
- 创建完整的数据库初始化脚本
- SQL 文件可重复执行，支持后续部署
- 运行测试验证 API 功能正常

**非目标：**
- 不创建测试数据（测试用例会自行创建）
- 不修改现有代码
- 不配置生产环境

## 决策

### 1. SQL 文件组织方式

**决策**：使用单一 `init.sql` 文件包含所有建表语句

**替代方案**：
- 每个表一个文件 → 过于分散，增加维护成本
- 使用迁移工具（如 golang-migrate）→ 对于初始化场景过于复杂

**理由**：简单直接，适合项目当前阶段

### 2. 表结构推断方式

**决策**：从 DAO 层代码反向推断表结构

**替代方案**：
- 从 models 层推断 → 可能遗漏数据库特定的约束和索引
- 询问用户提供 → 用户已说明需要从代码推断

**理由**：DAO 层的 SQL 语句是最准确的表结构来源

### 3. 主键生成策略

**决策**：使用 BIGINT AUTO_INCREMENT 作为主键

**替代方案**：
- 使用 UUID → 代码中使用的是 int64
- 使用雪花 ID → 需要应用层生成，但建表时仍用 BIGINT

**理由**：与现有代码保持一致

### 4. 测试执行方式

**决策**：进入 bluebell 容器内部执行 `go test -v ./...`

**替代方案**：
- 在宿主机执行 → 可能缺少依赖或配置不一致
- 逐个模块执行 → 不如批量执行高效

**理由**：容器内环境与服务运行环境一致，测试数据库与生产数据库隔离

## 风险 / 权衡

### 风险1：表结构与代码不匹配
**风险**：从 DAO 推断的表结构可能有遗漏
**缓解**：运行测试时会暴露问题，可快速修复

### 风险2：索引缺失影响性能
**风险**：可能遗漏某些索引
**缓解**：初始化脚本包含主要索引，后续可按需添加
