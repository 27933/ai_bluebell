## 新增需求

### 需求:展示用户列表
管理员必须能查看全站用户列表，支持分页，默认每页 20 条。

#### 场景:加载用户列表
- **当** 管理员访问用户管理页
- **那么** 必须调用 `GET /api/v1/admin/users` 并展示用户名、角色、状态、注册时间

#### 场景:按角色筛选
- **当** 管理员选择角色筛选项（all/admin/author/reader）
- **那么** 必须重新请求并只显示该角色的用户

#### 场景:按状态筛选
- **当** 管理员选择状态筛选项（all/active/inactive）
- **那么** 必须重新请求并只显示该状态的用户

### 需求:修改用户角色
管理员必须能将用户角色修改为 admin、author 或 reader。

#### 场景:修改角色成功
- **当** 管理员在用户行选择新角色并确认
- **那么** 必须调用 `PATCH /api/v1/admin/users/:id/role` 并刷新列表

#### 场景:修改自己的角色
- **当** 管理员尝试修改自己的角色
- **那么** 必须禁止操作并提示"不能修改自己的角色"

### 需求:封禁和解封用户
管理员必须能将用户状态切换为 active 或 inactive。

#### 场景:封禁用户
- **当** 管理员点击"封禁"并确认
- **那么** 必须调用 `PATCH /api/v1/admin/users/:id/status`（status: inactive）并刷新列表

#### 场景:解封用户
- **当** 管理员点击"解封"并确认
- **那么** 必须调用 `PATCH /api/v1/admin/users/:id/status`（status: active）并刷新列表

#### 场景:操作自己的账号
- **当** 管理员尝试封禁自己
- **那么** 必须禁止操作并提示"不能封禁自己"

### 需求:批量封禁用户
管理员必须能批量选择用户并一次性封禁或解封。

#### 场景:批量封禁
- **当** 管理员勾选多个用户后点击"批量封禁"
- **那么** 必须调用 `PATCH /api/v1/admin/users/batch/status`（status: inactive）并刷新列表
