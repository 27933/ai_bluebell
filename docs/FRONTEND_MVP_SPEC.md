## 🔐 用户角色与权限

### 用户角色

| 角色 | 权限 | 菜单项 |
|------|------|--------|
| 访客（未登录） | 浏览、搜索、查看评论 | 登录 / 注册 |

**说明**：
- 新注册用户默认为 Reader，需要管理员升级为 Author
- 角色升级后，下次登录生效

## 🔄 认证与Token管理

### Token机制

```
登录成功 → 存储 access_token + refresh_token

每个请求：
  ├─ 需要认证的接口 → Authorization: Bearer <access_token>
  └─ 公开接口 → 不需要token

Token过期处理：
  收到 1007（无效token）
    ↓
  使用 refresh_token 自动刷新
    ├─ 成功 → 更新本地token，重新发送原请求
    └─ 失败 → 弹出"登录已过期，请重新登录"提示，重定向到登录页
```

### 登录方式

**用户名 + 密码** 登录（不是邮箱）

```javascript
POST /auth/login
{
  "username": "testuser",
  "password": "test123456"
}
```

## 🎯 关键交互逻辑

### 1. 评论流程

**访客（未登录）**：
- 可以查看评论列表 ✓
- 不能发表评论

**已登录用户**：
- 可以查看评论列表 ✓
- 可以发表评论（如果 allow_comment=true）✓
- **发表评论后立即刷新列表** ✓

---

### 2. 点赞流程

**访客**：
- 不能点赞

**已登录用户**：
- 可以点赞/取消点赞
- **点赞后立即更新计数** ✓
- 幂等操作（重复点赞不报错）

---

### 3. 文章编辑流程

**新建文章**：
1. 进入编辑页面
2. 填写标题 + 内容（Markdown）
3. 可选：填写摘要、标签、SEO信息
4. 选择"保存草稿"（status=draft）或"发布"（status=published）
5. **Markdown编辑器需要用完整库**（如CodeMirror）

**编辑已有文章**：
1. 从仪表板点击"编辑"或从文章详情页点击"编辑"
2. 自动填充原文章信息
3. 修改并保存

**草稿自动保存**：
- 仅保存到**本地浏览器**（LocalStorage/IndexedDB）
- **不自动上传后端**
- 用户手动点击"保存草稿"或"发布"才上传

---

### 4. 错误处理

**所有错误统一用 Toast 展示**：

```javascript
错误情况 → Toast 提示 → 可选重试

示例：
- 参数错误 → "请填写所有必填项"
- 网络错误 → "网络连接失败，请重试"
- 权限错误 → "你没有权限执行此操作"
- 服务器错误 → "服务器错误，请稍后重试"
```

---

### 5. 分页

**用户可选**：
- 默认每页 20 条
- 用户可以在首页/搜索页选择每页显示数量
- 最大 50 条/页

---

### 6. 搜索功能

**简单搜索**：
- 首页搜索框 + 搜索按钮
- 进入搜索结果页面
- 支持按标签、排序、分页过滤

**不需要**：
- 搜索自动补全
- 搜索历史
- 热门搜索词

### 阅读趋势图

**使用 API**: `GET /article-stats/trend`

```
参数：
  - article_id: 文章ID
  - days: 7（一周）或 30（一月）
  - group_by: "day" 或 "week" 或 "month"

返回：
  {
    "trend": [
      {"label": "2026-04-01", "value": 180},
      {"label": "2026-04-02", "value": 220},
      ...
    ]
  }

前端：用这个数据绘制折线图
```

**图表库**：推荐 ECharts 或 Chart.js

## 🔄 后端需要修改的内容

### 1. 评论API权限修改

**当前**：`GET /comments` 需要 Authorization header

**修改**：允许不登录的用户（访客）也能查看

```
GET /comments?article_id=...
Authorization: (可选)

- 有token → 作为已登录用户处理
- 无token → 作为访客处理，直接返回评论列表
```

---

### 2. 创建文章权限验证

**确保后端**：`POST /articles` 时检查用户角色

```
POST /articles
Authorization: Bearer <token>

后端检查：
  ├─ 用户角色 === "author" → 允许
  └─ 用户角色 === "reader" → 返回 1013（无权限）
```

**前端配合**：reader 用户看不到"写文章"菜单项
