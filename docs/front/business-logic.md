# Bluebell 业务逻辑文档

> 供前端开发参考，描述核心业务流程、权限模型、状态机和交互规则

---

## 1. 认证与鉴权

### 1.1 Token 机制

```
登录成功 → 获得 access_token + refresh_token
    ↓
access_token 有效期 8 小时
    ↓
过期后 → 用 refresh_token 换取新 token 对
    ↓
refresh_token 也失效 → 重新登录
```

**前端实现建议：**

1. 登录后将 `access_token` 和 `refresh_token` 存储到 localStorage 或内存
2. 每个请求在 Header 中携带 `Authorization: Bearer <access_token>`
3. 收到 `1007`（无效 Token）时，尝试用 `refresh_token` 刷新
4. 刷新成功则替换本地存储的 token，重放失败的请求
5. 刷新失败（也返回 1007）则清除本地状态，跳转登录页
6. 收到 `1006`（需要登录）说明请求未携带 Token

### 1.2 用户角色

| 角色 | 说明 | 权限范围 |
|------|------|---------|
| visitor | 游客 | 仅浏览公开内容 |
| reader | 读者 | 浏览 + 评论 + 点赞 |
| author | 作者 | reader 权限 + 创建/管理文章、标签、栏目 |
| admin | 管理员 | 全部权限 + 后台管理 |

> 新注册用户默认角色为 `reader`。

### 1.3 接口权限分层

```
公开接口（无需 Token）
├── 文章列表/详情/搜索/精选/热门
├── 标签列表、标签下文章
├── 栏目列表/详情、栏目下文章
├── 作者主页/文章
├── 文章统计（日统计/趋势/批量）
├── RSS 订阅
└── 注册/登录/刷新 Token

登录接口（需要 Token）
├── 个人资料（查看/修改）
├── 文章 CRUD（仅自己的文章）
├── 评论 CRUD（仅自己的评论）
├── 点赞/取消点赞
├── 标签创建/修改/删除
├── 栏目创建/修改/删除（仅自己创建的）
├── 文件上传
└── 文章导出

管理员接口（需要 admin 角色）
├── 用户管理（列表/详情/状态/角色/批量）
├── 文章管理（列表/精选设置）
└── 系统监控（概览/日统计/实时指标/历史指标）
```

## 2. 文章生命周期

### 2.1 状态流转

```
             创建
              ↓
           [draft] ←──────────┐
              │                │
              ↓ 发布           │ 下线
         [published] ─────→ [offline]
              ↑                │
              └──── 重新发布 ──┘
```

| 状态 | 说明 | 前端展示 |
|------|------|---------|
| draft | 草稿 | 仅作者在"我的文章"中可见 |
| published | 已发布 | 所有人可见 |
| offline | 已下线 | 仅作者可见，不出现在公开列表中 |

### 2.2 文章创建流程

```
1. 用户填写 title（必填）、content（必填）
2. 可选填写 tags、slug、meta 信息
3. 选择保存为草稿（status=draft）或直接发布（status=published）
4. 后端自动：
   - 生成 Snowflake ID
   - 计算 word_count
   - 自动生成 summary（如果未提供）
   - 处理 tags（自动创建不存在的标签）
5. 返回完整文章对象（包含 id）
```

### 2.3 文章权限规则

| 操作 | 权限 | 说明 |
|------|------|------|
| 查看已发布文章 | 所有人 | 包括游客 |
| 查看草稿/已下线文章 | 仅作者 | 通过 `/author/articles` 接口 |
| 创建文章 | 已登录用户 | - |
| 编辑文章 | 仅作者 | 非作者操作返回权限错误 |
| 删除文章 | 仅作者 | - |
| 修改状态 | 仅作者 | 发布/下线 |
| 设置精选 | 作者或管理员 | - |
| 管理员管理文章 | 仅 admin | 通过 `/admin/articles` |

### 2.4 浏览量统计

- 访问文章详情接口会自动记录浏览
- 内置防刷机制（同一用户短时间内重复访问不重复计数）
- 也可通过 `POST /articles/view` 主动上报
- 统计数据有日统计、趋势分析、热门排行等维度

## 3. 评论系统

### 3.1 评论流程

```
1. 用户登录后，进入已发布文章的详情页
2. 如果 allow_comment=true，显示评论输入框
3. 提交评论：article_id（必填）+ content（必填）+ parent_id（可选）
4. 评论支持嵌套回复（通过 parent_id 实现）
5. 评论只有作者本人可以编辑/删除
```

### 3.2 评论权限

| 操作 | 权限 |
|------|------|
| 查看评论列表 | 需要登录 |
| 创建评论 | 需要登录 |
| 编辑评论 | 仅评论作者 |
| 删除评论 | 仅评论作者 |

> **注意：** 获取评论列表（`GET /comments`）也需要登录，前端需在请求中携带 Token。

## 4. 点赞系统

### 4.1 点赞逻辑

```
点赞操作（POST /likes）：
  ├── 文章点赞：target_type=article, target_id=文章ID
  └── 评论点赞：target_type=comment, target_id=评论ID

取消点赞（DELETE /likes）：
  └── 通过查询参数传递 target_type 和 target_id

查询状态（GET /likes/status）：
  └── 返回 is_liked（是否已赞）和 like_count（总赞数）
```

### 4.2 幂等性

- **重复点赞**：不会报错，返回成功（幂等）
- **取消未点赞的内容**：不会报错，返回成功（幂等）
- 前端无需做额外的重复检查，直接发请求即可

### 4.3 前端交互建议

```
1. 进入文章/评论页面时，调用 GET /likes/status 获取点赞状态
2. 点击赞按钮：
   a. 如果 is_liked=false → POST /likes → 切换为已赞状态，like_count+1
   b. 如果 is_liked=true → DELETE /likes → 切换为未赞状态，like_count-1
3. 可先乐观更新 UI，再等接口返回确认
```

## 5. 标签系统

### 5.1 标签特点

- 标签是全局共享的，所有用户创建的标签都在同一个标签池中
- 创建文章时传入 `tags: ["Go", "教程"]`，后端会自动创建不存在的标签
- 标签包含 `article_count` 字段，表示使用该标签的文章数
- 标签列表 API 直接返回数组（不是 `{ list: [] }` 格式）

### 5.2 作者标签

`GET /author/tags` 返回当前用户文章中使用过的标签，适合在写作界面做标签推荐。

## 6. 栏目系统

### 6.1 栏目与文章关系

```
栏目(Category) ←──── 多对多 ────→ 文章(Article)

一篇文章可以属于多个栏目
一个栏目可以包含多篇文章
```

### 6.2 栏目操作流程

```
创建栏目 → 创建文章 → 将文章添加到栏目
                         ↓
              POST /articles/:id/categories
              body: { "category_ids": [1, 2] }
```

### 6.3 栏目权限

| 操作 | 权限 |
|------|------|
| 查看栏目列表/详情 | 所有人 |
| 查看栏目下文章 | 所有人（仅返回已发布文章） |
| 创建栏目 | 需要登录 |
| 编辑栏目 | 仅创建者或管理员 |
| 删除栏目 | 仅创建者或管理员 |
| 添加文章到栏目 | 仅文章作者 |

## 7. 作者主页

### 7.1 公开访问

通过 `GET /authors/:username` 获取作者公开信息，包含：
- 昵称、简介
- 加入日期
- 文章数、总浏览量、总赞数

通过 `GET /authors/:username/articles` 获取作者已发布的文章列表。

### 7.2 与个人中心的区别

| | 作者主页 | 个人中心 |
|--|---------|---------|
| 接口 | `/authors/:username` | `/auth/profile` |
| 权限 | 公开 | 需要登录 |
| 数据 | 仅公开信息 | 包含 email 等私有信息 |
| 文章 | 仅已发布 | 所有状态 |

## 8. 文件上传

### 8.1 上传方式

使用 `multipart/form-data` 格式，字段名为 `file`。

### 8.2 上传后使用

上传成功后返回 `url` 字段，可直接用于：
- 文章内容中引用图片（Markdown `![](url)`）
- 用户头像设置（更新个人资料的 `avatar` 字段）

## 9. 管理后台

### 9.1 功能模块

```
管理后台
├── 文章管理
│   ├── 查看所有文章（含所有状态）
│   └── 设置/取消精选
├── 用户管理
│   ├── 用户列表（按角色/状态筛选）
│   ├── 用户详情
│   ├── 修改用户状态（启用/禁用）
│   ├── 修改用户角色
│   └── 批量修改用户状态
└── 系统监控
    ├── 数据概览（用户数/文章数/评论数/今日新增）
    ├── 日统计趋势（按日期范围）
    ├── 实时性能指标（CPU/内存/磁盘/活跃用户）
    └── 性能历史（按时间范围和指标类型）
```

### 9.2 权限验证

管理员接口使用双重中间件验证：
1. `JWTAuthMiddleware` — 验证登录状态
2. `AdminAuthMiddleware` — 验证 admin 角色

普通用户访问管理员接口不会返回 1006（需要登录），而是返回权限错误（1013 或非 1000 的其他错误码）。

## 10. 分页规范

### 10.1 请求参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| page | int | 1 | 页码，从 1 开始 |
| size | int | 20 | 每页条数 |

### 10.2 响应格式

```json
{
  "data": {
    "list": [...],
    "total": 100,
    "page": 1,
    "size": 20,
    "pages": 5
  }
}
```

### 10.3 特殊情况

- 部分接口（如标签列表 `GET /tags`）直接返回数组，不使用分页格式
- 栏目列表只返回 `{ "list": [...] }`，没有 `total` 和 `pages`
